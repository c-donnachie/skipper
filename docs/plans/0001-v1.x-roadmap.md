# Plan 0001 — v1.x roadmap and validation

- **Status**: Active
- **Created**: 2026-04-30
- **Related PRD**: docs/prds/0001-private-plugin.md, 0002-rico-plugin.md, 0003-more-stacks.md
- **Related ADR**: docs/decisions/0001-monolithic-vs-multi-plugin.md

## Context

skipper hit v1.0.1 (i18n + submission-ready) on 2026-04-29. The feature set is stable: 17 skills, 9 subagents, 8 stacks, 6 layers, 3 hooks. The official Anthropic marketplace submission is the next external milestone (see plan 0002).

Before pushing more features, we need real-world validation: use v1.0.1 intensively for 1-2 weeks, collect feedback (own usage + friend testing), and iterate on UX issues before scaling to add-ons.

## Steps

### Phase 1 — Validation (1-2 weeks)

- [ ] Use skipper daily in `not-pato` (Expo+Supabase) — primary test bed.
- [ ] Try skipper in 1-2 other projects with different stacks (one web, one backend if possible).
- [ ] Friend testing: share v1.0.1 with at least 2 collaborators, collect honest feedback.
- [ ] Capture issues in GitHub: separate by category (bug, UX, feature request, doc gap).
- [ ] After 1 week, audit: which issues are critical vs cosmetic?

### Phase 2 — v1.0.x patches (as issues come in)

- [ ] Fix critical bugs found during validation.
- [ ] Translate `suggest.sh` Stop hook output to English (currently still in Spanish).
- [ ] Add `.claude/.skipper-*` to a `.gitignore` template applied during `init-structure`.
- [ ] Capture screenshots/GIFs for the README (still has TODO placeholders).
- [ ] Push v1.0.2, v1.0.3, etc. as needed.

### Phase 3 — v1.1+ feature work (only after validation is solid)

Priority order based on validation feedback:

- [ ] **More stacks** (PRD-0003): start with Astro+Supabase if any user requests it.
- [ ] **Private plugin** (PRD-0001): build as separate plugin in madagascar.
- [ ] **Rico plugin** (PRD-0002): build as separate plugin if there's demand for cross-file refactors.

### Phase 4 — Marketplace submission

Tracked in plan 0002.

## Critical files

- `docs/decisions/` — when bringing new features, write ADRs FIRST.
- `lib/detect.sh` — when adding stacks, this is the central place to update.
- `lib/test-detect.sh` — every new stack needs a passing test fixture.
- `CHANGELOG.md` — every release bumps here.
- `examples/fixtures/<new-stack>/` — minimum marker files for new stacks.

## Verification

Phase 1 done when:
- 1-2 weeks of personal usage logged.
- At least 5 distinct issues filed in GitHub from real usage (mix of bug/UX/feature).
- Friend feedback collected (informal is fine — not a UX research study).

Phase 2 done when:
- All critical issues from phase 1 have a corresponding patch released.
- README screenshots are no longer placeholders.

Phase 3 done when:
- At least 1 new stack OR 1 add-on plugin is released.

## Notes

- **Don't rush to v2.0.** v1.x can run for months. Most projects in this category (frameworks, CLIs) live on v1.x for 1-2 years before bumping major.
- **Resist feature creep**: each new feature should have a PRD first. ADRs document the decisions, PRDs document the user-facing scope.
- **Submission is parallel**: don't block validation on Anthropic's review. Both happen at the same pace.
